## Context

当前项目需要一个统一日志方案来替代散落在多处的 `std::cout` / `printf` 调试输出。基础 logger 应该提供控制台和文件输出能力，支持运行时实时落盘，并为后续日志级别控制、日志格式扩展以及持久化增强提供基础。

现有代码库结构中已有 CMake 目标和单元测试基础，新增 logger 时应保持依赖简单，优先采用标准 C++ 接口实现。

## Goals / Non-Goals

**Goals:**
- 提供基础 logger 接口和实现，支持 `debug`、`info`、`warning`、`error` 等日志级别
- 支持控制台输出，并支持运行时配置文件路径，实现实时落盘
- 记录日志时附带时间戳、级别、可选模块名
- 在并发场景下保证单条日志完整输出，不出现行内容交叉污染
- 为每条日志增加进程 ID 和线程编号上下文
- 提供简单的初始化和关闭接口，避免多次打开文件或资源泄漏
- 通过单元测试覆盖日志格式、文件写入、级别过滤以及多线程输出行为

**Non-Goals:**
- 不实现复杂日志轮转、异步日志、网络日志或结构化 JSON 日志
- 不支持多进程共享同一日志文件
- 不提供跨平台的高级日志后端适配（只需支持标准 C++ 文件与控制台输出）

## Decisions

- 使用标准 C++ `std::ofstream` 和 `std::ostream` 实现日志写入，避免额外依赖。
- Logger 采用静态接口或单例样式，方便全局调用，但内部应保证资源安全和可控初始化顺序。
- 运行时落盘通过配置接口启用文件路径，初始化后所有日志同时写入控制台和文件。
- 日志级别由枚举控制，低于当前级别的消息直接丢弃。
- 日志格式扩展为可读字符串格式，至少包含：时间戳、级别、进程 ID、线程编号，以及可选模块名。
- 并发日志写入采用串行化输出策略，保证单条日志原子写出；日志顺序定义为 logger 接收并提交输出的顺序，而非跨线程真实因果顺序。
- 线程编号采用 `thread_local` 缓存的单调递增整数，在每个线程首次记录日志时通过全局原子计数器分配，格式统一为 `thread=<n>`。
- 在 `src/main.cpp` 中提供示例用法，展示控制台输出与文件落盘功能。

## Risks / Trade-offs

- 选择同步写文件实现简单易用，但在高频日志场景下可能影响性能；如果后续需要可考虑改为异步写入。
- 不使用外部日志库降低依赖，但也牺牲了现成功能（如日志轮转、压缩、格式化插件）。
- 静态接口简化使用，但在多次初始化或多模块测试中需要谨慎管理全局状态，并在并发首次初始化时避免竞态。
- 引入线程编号要求后，需要定义编号来源和格式稳定性，否则日志格式会不稳定且不利于排查。
- 文件写入失败时需要明确错误处理策略，避免丢失日志或导致程序异常。
